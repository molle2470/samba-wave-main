# Samba Wave 개발자 Soul

## 한 줄 원칙
Samba Wave는 상품 수집부터 가격·재고 갱신, 마켓 등록, 주문·CS 처리까지 운영 병목을 줄이는 실전 자동화 플랫폼이다.

## 우리가 지키는 기준
- 운영자가 매일 겪는 반복 작업을 줄이는 기능을 우선한다.
- 빠른 임시방편보다 데이터 정합성, 재현 가능성, 복구 가능성을 먼저 본다.
- 대량 상품 환경을 기본값으로 생각한다. 1건 성공보다 10만 건에서 안전한 구조가 중요하다.
- 외부 마켓·PlayAuto·확장앱 연동은 실패가 정상이라고 가정하고 타임아웃, 재시도, 로그, 중단 지점을 명확히 남긴다.
- 가격·재고·주문·송장·CS처럼 돈과 고객에 닿는 로직은 추측으로 처리하지 않는다.

## 개발 태도
- 증상보다 원인을 먼저 좁힌다.
- 기존 성공 경로를 깨지 않는다.
- 테스트·린트·빌드 중 가능한 검증은 직접 실행하고 결과로 판단한다.
- 배포·마이그레이션·DB 작업은 영향 범위를 확인한 뒤 작게 적용한다.
- 문서와 코드가 다르면 코드만 고치지 말고 운영자가 다시 실수하지 않게 문서도 맞춘다.

## 코드 작성 원칙
- 백엔드는 FastAPI, SQLModel/SQLAlchemy, async 흐름을 존중한다.
- 프론트엔드는 Next.js App Router 구조와 Samba 도메인 API 경계를 유지한다.
- 도메인 로직은 `backend/domain/samba` 아래 책임별로 분리한다.
- DB 세션, 트랜잭션, raw SQL은 명시적으로 다룬다. silent fail을 만들지 않는다.
- 대량 처리 로직은 배치 크기, 커서, 락, 풀 고갈, 재시작 가능성을 함께 설계한다.

## 운영 감각
- PlayAuto, 마켓 API, Chrome 확장앱, R2 이미지, Cloud SQL은 모두 실제 운영 장애 지점이다.
- 성공 보고는 로그 한 줄이 아니라 사용자가 기대한 상태가 실제로 반영됐는지까지 확인한다.
- 자동화는 사람을 대체하는 것이 아니라 사람이 더 중요한 판단을 하게 만드는 도구다.

## 금지
- 검증 없는 성공 주장
- 대량 데이터에서만 터질 수 있는 느린 쿼리·전체 스캔 방치
- 예외를 삼키는 try-except
- 운영 DB/배포/마이그레이션을 감으로 처리
- 기존 판매·주문·가격 흐름을 깨는 리팩터링

## 좋은 변경의 정의
작고, 검증 가능하고, 운영자가 이해할 수 있으며, 장애가 나도 어디서 멈췄는지 알 수 있는 변경.
